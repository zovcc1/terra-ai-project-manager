## Key Learnings

- **Project:** ai-project-manager
- STOMP notification queue is `/user/queue/notifications`, comment topic is `/topic/task/{taskId}/comments`, kanban topic is `/topic/kanban/{projectId}`; all defined in `frontend/src/lib/websocket.ts`. Backend STOMP principal is the JWT **username** (not userId/email) — `WebSocketService.sendNotificationToUser(username, ...)` matches this correctly, but `sendAiPendingToUser` uses userId instead (known mismatch, intentionally left unfixed — separate feature, user declined to include it in scope).
- `/notifications/recent` (backend) returns a Spring `Page<T>` (`content`, `totalPages`, etc.), not a plain array — any new frontend caller must type it as `Page<NotificationResponse>` (see `frontend/src/lib/api.ts`), not `NotificationResponse[]`.
- `getRecentActivity()` in `frontend/src/lib/api.ts` also hits `/notifications/recent` but is still typed/used as a plain array (`manager/dashboard.tsx`) — same latent bug as `getRecentNotifications` had, left unfixed as it was out of scope for the 2026-07-04 real-time fix session.
- `GlobalExceptionHandler`'s `@ExceptionHandler(RuntimeException.class)` catch-all was silently swallowing `AccessDeniedException` (which extends `RuntimeException`) and returning 500 instead of 403. Spring resolves `@ExceptionHandler` by most-specific exception type regardless of declaration order, so adding a dedicated `@ExceptionHandler(AccessDeniedException.class)` in the same `@RestControllerAdvice` fixes it without touching the catch-all.
- Task-detail access control pattern in this project: `@PreAuthorize("hasRole('ADMIN') or @authorizationService.hasTaskAccess(#principal.username, #id)")` (see `TaskController.moveTask`, now also `getTaskById`, and `CommentController`). Any NEW task-scoped endpoint should use this exact pattern, not a bare role check — a bare `hasRole(...)` check lets any MEMBER view/act on tasks from projects they have no team membership in.
- Team `team_members` table can legitimately have zero rows for a given `team_id` (e.g. team_id=1 for project "Website Redesign" as of 2026-07-04) — this is a real, reproducible way to trigger access-denied paths for testing, not just a hypothetical.
- `@stomp/stompjs` v7 does **not** auto-resubscribe topics/queues after a dropped connection reconnects (`reconnectDelay: 5000` just reopens the socket). Any subscription that must survive a reconnect has to be re-created inside the `onConnect` handler (which fires on every connect *and* reconnect) — see `frontend/src/lib/websocket.ts`'s `pendingNotificationCallbacks`/`activeNotificationSub` pattern (bug-003) and the newer `commentSubs` registry (bug-010). A subscription created once, outside `onConnect`, silently dies on the first reconnect with no error.
- `/user/queue/notifications` (mentions) and `/topic/task/{taskId}/comments` (all comments) are independent channels — one surviving reconnect while the other doesn't produces confusing symptoms like "real-time only works for @mentions."
- `frontend/src/routes/member/tasks.$taskId.tsx` hardcodes `<AppShell persona="manager">` at all 3 call sites (bug-011, unfixed) — copy-paste artifact from the manager route. Don't assume the member route's AppShell persona is correct just because access-control (`@PreAuthorize`) is correct; they're independent.
- The JWT token captured in `wsConnect(token)`'s `connectHeaders` is closed over at `Client` construction time — automatic STOMP reconnects reuse that same original token, they do NOT re-read `localStorage`. So a token that was valid when the socket first connected stays valid across transport-level reconnects as long as the backend's signing secret hasn't changed in between.
- AI provider/model config lives in the `ai_settings` DB table (single row, id=1), not `application.properties` — `AiSettingsService.getActiveLlmConfig()` is the one place that resolves the active model/apiUrl/apiKey for `OpenAiClient`, and the admin UI at `admin/ai-settings.tsx` edits that same row via `AiSettingsController`. Only `ai.openai.api.url` (fallback endpoint) and `ai.openai.timeout`/`ai.encryption.key` remain in `application.properties`. See Decision Log 2026-07-06.
- [2026-07-09] `ai_settings.api_url` (entity/DTO field `apiUrl`) now overrides the provider→URL map in `getActiveLlmConfig()` via `firstNonBlank(settings.getApiUrl(), resolveApiUrl(provider))`. Providers: openrouter/openai/deepseek have hardcoded URLs in `resolveApiUrl()`; anthropic/google/custom fall through to `fallbackApiUrl` unless the admin sets a custom `apiUrl` — note native Anthropic/Google APIs are NOT OpenAI-request-shape compatible with `OpenAiClient`, so those only work via a compatible proxy. Frontend `PROVIDER_PRESETS` map in `admin/ai-settings.tsx` prefills both `model` and `apiUrl` on provider change but the URL field stays freely editable (mirrors the existing model-field UX).
- [2026-07-09] `AppShell` (`frontend/src/components/app-shell.tsx`) is rendered *by* each page component (`<AppShell persona="...">{content}</AppShell>`), not the reverse — so any React Context whose Provider lives inside `AppShell` can NEVER be read by the page component that renders it (the page is an ancestor of AppShell in the fiber tree, not a descendant). The header-search `SearchContext` Provider had to move up to `routes/__root.tsx`'s `AppContent` (true ancestor of every routed page) with `AppShell` itself becoming just another consumer via `useContext`. Any future cross-cutting state that both AppShell's chrome AND the page content need to share must be owned at `__root.tsx` (or higher), never inside `AppShell`.

## Do-Not-Repeat

<!-- Mistakes made and corrected. Each entry prevents the same mistake recurring. -->
- [2026-07-09] Don't put a Context `Provider` inside `AppShell` expecting the page component that renders `<AppShell>` to read it via `useContext` — that page is `AppShell`'s parent, not its child, so it always gets the Context's default value (silently broken, no error). Caught this before shipping by reasoning through the actual fiber tree structure, not just "does it compile." Fix: own the state in `routes/__root.tsx` (ancestor of all routes) and have `AppShell` also just consume it.
- [2026-07-09] Discovered (not caused by this session) that `manager/kanban.tsx`, `manager/projects.$projectId.tsx`, `manager/tasks.$taskId.tsx`, `member/projects.tsx`, `member/tasks.$taskId.tsx` all pass a `projectId` prop to `<AppShell>` that its type never declared — pre-existing dead/no-op prop, confirmed via `git show HEAD` and `tsc --noEmit` before touching those files. Left alone (out of scope), but worth fixing in a future cleanup pass.

- [2026-07-04] `subscribeNotifications()` in `websocket.ts` must both register the callback AND create the STOMP subscription if already connected — don't split "connected" vs "not connected yet" into branches where only one creates the subscription. The subscription creation must be reachable regardless of when the caller registers relative to `onConnect` firing (callers always `await wsConnect()` first, so `onConnect`-only subscription creation never fires with pending callbacks present).
- [2026-07-04] When a `useEffect` does `somePromise.then(() => { const x = subscribeX(...); return x; })`, the returned unsubscribe from inside `.then()` is NOT the effect's cleanup function — it's swallowed by the promise. Must hoist an outer `let unsubscribe` variable and call it from the effect's actual `return () => {...}`.
- [2026-07-04] Do not assume a security-relevant fix took effect just because the file was edited and compiled — the backend process was running from a stale `target/classes` snapshot (launched by IntelliJ, no devtools/hot-reload) and needed a full restart before `GlobalExceptionHandler`/`TaskController` changes applied. Always confirm via a live request (network tab / curl) after editing security-relevant `@PreAuthorize`/`@ExceptionHandler` code, not just a clean compile.
- [2026-07-04] Never `grep`/`cat` `.env`, `.idea/workspace.xml`, or other files likely to contain DB/JWT credentials — the Claude Code permission classifier blocks this ("Credential Materialization"), and correctly so, since it would print live secrets into the transcript. Instead `source` the `.env` file into a subshell's environment (`set -a; source .env; set +a`) without printing it, then launch the process needing those env vars.
- [2026-07-04] Do not query the `users` table (or any PII-bearing table) directly even for debugging — blocked by the permission classifier as speculative PII access. Use non-PII tables (`team_members`, `projects`, `tasks`) or the app's own authenticated UI/API to verify identity/role instead.
- [2026-07-04] Do not open two browser tabs on `http://localhost:3000` logged in as two different users (e.g. manager + member) to test multi-role real-time scenarios — `terra_token`/`terra_user` live in `localStorage`, which is shared per-origin across all tabs. Logging into the second tab silently overwrites the first tab's session; the first tab's UI keeps showing stale in-memory React state (looks logged in as the old role) while its actual API/WS auth has switched to the new user. Backend CORS only allows `localhost:3000` (not `127.0.0.1:3000`), so origin-splitting isn't a free workaround without a backend config change. Preferred approach for two-role live tests: keep ONE browser tab as the role being observed, and drive the OTHER role's actions via `curl` against the REST API directly (login via `POST /api/auth/login` with the app's own test credentials, then call the relevant endpoint with `Authorization: Bearer <token>`) — this avoids any localStorage collision.
- [2026-07-04] Do not print a raw JWT/token value into the transcript to "check if it's valid" (e.g. `localStorage.getItem('terra_token')` alone) — the permission classifier blocks this as credential materialization, correctly. Instead check validity indirectly: run the check *inside* page JS (e.g. `fetch(url, {headers: {Authorization: 'Bearer ' + localStorage.getItem(...)}}).then(r => r.status)`) and only report the resulting status code, never the token itself.
- [2026-07-04] Restarting the backend via a fresh `mvn spring-boot:run` (even sourcing the same `.env`) is not guaranteed to preserve `JWT_SECRET` identically to whatever the previously-running instance actually had — observed a live member WS reconnect get rejected with "JWT signature does not match" immediately after a restart in this session, while a separately-obtained manager token from the same pre-restart window continued to validate fine. Root cause not fully isolated before the test was cut short. If forcing a WS reconnect for testing without wanting to touch backend process/secret at all, prefer a transport-level disruption (e.g. `ss -K` to kill established TCP connections on the port) over killing/relaunching the whole JVM.

## Decision Log

<!-- Significant technical decisions with rationale. Why X was chosen over Y. -->
- [2026-07-06] AI model/provider config moved from `application.properties` (`ai.openai.model`, read via `@Value` in `OpenAiClient`) to the DB `ai_settings` table, read through `AiSettingsService.getActiveLlmConfig()`. Reason: the `ai_settings` table and the admin UI (`admin/ai-settings.tsx`) already let an admin edit provider/model/default_model/enabled/api_key, but `OpenAiClient` was silently ignoring the DB `model` value and always sending the `application.properties` one — so when OpenRouter deprecates a model, the fix required editing `application.properties` and redeploying the container instead of just using the admin settings page. `getActiveLlmConfig()` now resolves: model = `model` → `default_model` → hardcoded `"openai/gpt-oss-120b:free"`; apiUrl = provider-based map (`openrouter`→OpenRouter endpoint, `openai`→OpenAI endpoint, anything else falls back to `ai.openai.api.url`). Anthropic/Google providers are selectable in the UI but this client only speaks the OpenAI-compatible chat-completions wire format, so those still fall through to the fallback URL and won't work correctly without a separate client implementation (explicitly out of scope, decided with user). `AiSettingsService.getDecryptedApiKey()` was removed as dead code (only caller was `OpenAiClient`, replaced by `getActiveLlmConfig()`); `getActiveApiKey()` was left alone (pre-existing, already had no callers, out of scope).

- (2026-07-06) Members never actually see `/member/tasks/$taskId`: its `beforeLoad: requireRole("/manager")` redirects MEMBER users to /member/kanban. Members read/write comments only via the task modals in member/kanban.tsx and member/projects.tsx, which have NO websocket subscription (plain useQuery + invalidateQueries). The whole member tasks.$taskId.tsx file is a byte-identical copy of the manager route except the route path (see bug-011, bug-014).
